# v3 经济系统设计文档（负债 + NTR 安慰币）

> 两个相互独立但都影响金币流的设计：负债经济（可负余额、-500 硬封顶、不累积利息）+ NTR 安慰币（攻击者零代价，被牛方按稀有度收补偿）。
> 日期：2026-07-12（v3.1 已根据用户拍板重新设计）
> 版本：v3.1 经济系统 beta
> 关联文档：`v3_离婚系统.md`（已批准）·`v3_接力战_主线.md`（已批准）·`玩法升级策划案.md`（v2 基线）

## [S1] 一句话描述

**负债经济**：余额可以为负，最低 -500 币硬封顶；欠债时所有产出优先抵债；不累积利息（用户决策"做得简单"）。
**NTR 安慰币**：NTR 成功后攻击者**不付任何手续费**，被牛方收到"安慰币"（按被偷老婆稀有度 + 好感度计算，等同于离婚返还基础）。

## [S4] 负债经济机制

### [S4.1] 触发与下限

```python
DEBT_FLOOR = -500  # 硬封顶，玩家余额永远 ≥ -500
```

**触发场景**（任何扣款超过余额时）：

| 扣款 | 触发负债的场景 |
|------|--------------|
| 离婚分家产正分走 | 余额 100，分走 30 → -30（负债） |
| NTR 失败赔偿 | 余额 50，失败扣 10 → -40（负债）|
| 商城购买 | ❌ 不允许（负债时拒绝购买）|
| 打工启动费 | ❌ 不允许（负债时拒绝启动） |
| 复仇失败 | Phase 4 已有"失败 10 币安慰"——可能扣到负债 |

### [S4.2] 0 利息模式（用户决策）

```python
# 不累积利息（保持简单）
# 余额 -100 就一直是 -100，不增长、不减
```

**简化合理性**：
1. 避免指数债务
2. 避免每日结算代码
3. 教育玩家负债经济
4. 通过打工/离婚/NTR 被牛安慰币自然还债

### [S4.3] 优先抵债规则

```python
def apply_income(current_coins: int, income: int) -> int:
    if current_coins >= 0:
        return current_coins + income        # 无债，直接入账
    if current_coins < 0:
        deficit = -current_coins
        if income >= deficit:
            return current_coins + income    # 完全填平 + 剩余入账
        else:
            return current_coins + income    # 全部还债，余额仍负
```

### [S4.4] 触底保护

```python
def try_deduct(current_coins: int, amount: int) -> tuple[bool, int]:
    """扣款后余额不能 < DEBT_FLOOR (-500)。"""
    if current_coins - amount < DEBT_FLOOR:
        return False, current_coins
    return True, current_coins - amount
```

### [S4.5] 联动限制

| 行为 | 负债状态下 | 说明 |
|------|---------|------|
| **打工** | ❌ 不能启动 | 启动费必须从非负余额走 |
| **商城购买** | ❌ 不能买 | 余额必须 ≥ 0 |
| **PK** | ✅ 可参与 | 胜方币进还债桶 |
| **NTR 攻击** | ✅ 可发起 | 不扣手续费，**攻击者不可能因 NTR 触发负债** |
| **离婚** | ✅ 可发起 | 关键减压出口（双向百分比减压）|
| **复仇** | ✅ 可发起 | 免费 |
| **抽卡**（用券） | ✅ 可抽 | 预付费 |
| **抽卡**（用币） | ❌ 不能抽 | 单抽券 30 币需正余额 |
| **签到 / 任务** | ✅ 自动领取 | 进还债桶 |

## [S5] NTR 安慰币机制（撤掉手续费 · Q-D/Q-E）

### [S5.1] 核心决策（Q-D 用户原话）

> **"NTR 就是把别人老婆抱回家，付什么手续费呀？"**
> **"有个安慰币算好了"**

**结论**：
- **攻击者**：NTR 成功 = 仅获得老婆，不扣任何金币
- **被牛方**：收到"安慰币"，作为失去老婆的部分补偿
- **没有手续费 → 没有系统池 → 没有双向结算**

### [S5.2] 安慰币公式（与离婚返还基础对齐 · Q-E）

```python
# 安慰币 = 离婚返还基础（保持两个经济系统对称）
COMFORT_BASE = {
    "N":    10,
    "R":    25,
    "SR":   60,
    "SSR": 120,
}

def calc_ntr_comfort(target_rarity: str, target_intimacy: int) -> int:
    """NTR 安慰币 = 离婚返还基础 × (1 + 好感度 / 200)

    设计意图：安慰币 ≈ "如果离婚，你本可以拿到的钱"
    """
    base = COMFORT_BASE[target_rarity]
    mult = 1 + target_intimacy / 200.0      # 100 好感 → ×1.5
    return int(base * mult)
```

| 老婆 | 基础 + 好感倍率 | NTR 安慰币 |
|------|--------------|---------|
| N + 0 好感 | 10 × 1.00 | 10 币 |
| R + 50 好感 | 25 × 1.25 | 31 币 |
| SR + 80 好感 | 60 × 1.40 | 84 币 |
| **SSR + 100 好感** | **120 × 1.50** | **180 币** |

### [S5.3] 攻击者无任何操作

```python
def try_ntr(attacker_uid, target_wid) -> NtrResult:
    # ... 现有 NTR 概率流水线（攻方: 锁定/元素克制等不变）...
    if ntr_succeeds:
        # 1. 老婆转移（标准逻辑）
        # 2. 攻击者：不扣任何币（用户决策 Q-D）
        # 3. 被牛方收安慰币
        comfort = calc_ntr_comfort(target_rarity, target_intimacy)
        economy.earn(target_uid, comfort, reason="ntr_comfort")
        return NtrResult(success=True, attacker_paid=0, victim_received=comfort)
    else:
        # 攻击者：失败不扣币（用户决策 Q-B 沿用）
        return NtrResult(success=False)
```

### [S5.4] 与离婚系统的镜像关系

| 行为 | 离婚（主动放弃）| NTR（被动失去） |
|------|-----------|--------------|
| 触发 | 自己 | 对方 |
| 返还/补偿方向 | 自己拿币 | 被牛方拿币 |
| 来源 | 老婆"价值"换算 | 系统发币（无扣对手）|
| 是否扣对方币 | ❌（除非分家产触发）| ❌ **用户决策不扣任何币** |
| 概率惩罚 | 有（好感度 → 分家产概率↑）| 无 |

> NTR 安慰币和离婚返还**完全平行但不重叠**——离婚是玩家主动选择的高风险高回报，NTR 是被动损失但有补偿。

### [S5.5] 复仇免费（沿用）

- 复仇成功：仅老婆转移，无任何币结算
- 复仇失败：10 币安慰（Phase 4 沿用）
- **复仇路径不需要任何经济系统改动**

- 但发起 NTR 失败时会被复仇（如果被复仇）
- 复仇不收费，被复仇者可以利用这个机会

## [S13] 经济沙盒演算

### 案例 1：负债玩家日常
```
  Day 0: 余额 0 → 抽卡 -50 → 余额 -50（负债）
  Day 1: 余额 -50（无利息）
  Day 1: 签到 +15 → 抵债 → 余额 -35
  Day 2: 打工被拒（启动费需正余额）
  Day 2: 离婚 N 卡 (返还 10) → 还债 → 余额 -25
  Day 7: 余额 -25（无利息不增长）
```

### 案例 2：被牛救济（关键路径）
```
  攻击方 @xxx：余额 600
  NTR 守方 @负债 -300 的 SSR + 100 好感：
    攻击方：0 币（零代价）
    被牛方：+180 币（安慰币）
  → @负债 -300 → -120（救济 180 币，但仍负）
```

### 案例 3：连发 NTR 不破产（攻击者零代价验证）
```
  @xxx：余额 500
  NTR 1：成功 → 余额 500（不变）
  NTR 2：成功 → 余额 500（不变）
  NTR 3：失败 → 余额 500（不变）
  NTR 4：成功 → 余额 500（不变）
  
  → N 次 NTR 后余额始终 500，证明攻击者零代价
```

### 案例 4：离婚 + 被牛 经济对冲
```
  用户：余额 -300（负债），持有 SSR ❤️ 80
  选项 A：离婚 SSR+80 → 返还 60×1.4 = 84 币
          P(分家产 50%):  触发 = -240（负债 → -120 + 返还 84 = -36 币）
                          不触发 = -216（负债 + 84 = -216 币）
  
  选项 B：留着等被 NTR → 等待中可能被复仇 / 可能被牛
          被牛 (SSR+80) → 收到 60×1.4 = 84 币安慰币
                          （跟选项 A 不触发分家产时几乎一样）
  
  → 玩家行为：要么主动离婚高风险（可能分家产），要么等被牛拿安慰币（无风险）
  → 这是合理的双路径选择
```

## [S15] 设计自审

### Q：NTR 攻击者零代价会不会让所有人乱 NTR？

A：不会（由其他机制约束）：
1. **每日 NTR 次数限制**（Phase 4 沿用）
2. **被打后保留记录**——可能被复仇
3. **心理成本**——群里名声会被记住（evil_points 系统）

### Q：被牛收到 180 币会不会让养 SSR 无风险？

A：不会：
- 180 < 240（离婚时拿到的全额），所以"被 NTR" 比"主动离婚"少拿 60 币
- 还有"复仇窗口失败"的负向情绪
- 仍有复仇令牌机制（Phase 4 沿用）

### Q：负债玩家被牛会不会刷分？

A：不会（攻击者无代价 ≠ 攻击者赚钱）：
- 攻击者拿到老婆（已有机制）
- 被牛方拿到安慰币（救济）
- 系统没新增产出，不会通胀

### Q：为什么不收手续费了？

A：用户原话："NTR 就是把别人老婆抱回家，付什么手续费呀"。这跟现实语义一致——偷东西的人不向对方付钱，被偷的人可能拿点保险赔付。
