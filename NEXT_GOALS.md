# AnimeWifeX Next Goals

> 目的：基于当前代码现状，整理“还没做完什么”和“下一步最值得做什么”，方便继续排期。

## 1. 当前结论

当前插件已经具备可玩的主流程：

- 抽老婆 / 十连
- 查老婆 / 面板 / 图鉴
- 牛老婆 / 复仇 / 锁定
- 换老婆 / 交换老婆
- PK（已支持按编号指定双方出战老婆）
- 打工（已支持按编号指定老婆）
- 经济 / 商城 / 背包 / 任务 / 签到
- 管理员测试与重置命令

但从“玩家体验是否完整”和“策划预留是否真正落地”两个维度看，仍有一批明确待办。

## 2. 明确未完成项

### 2.1 仍未开放的玩家命令

当前还保留占位：

- `老婆 列表`
- `老婆 查`

这两个命令现在仍返回未开放提示，真实可用入口还是 `查老婆`。

### 2.2 打工第二波玩法仍是预留，未接线

代码里已有预留字段或配置，但尚未形成完整玩法：

- `work_contract_cost`
- `work_contract_reward_multiplier`
- `work_partner_bonus`
- `work_partner_daily_limit`
- `intimacy_decay`
- `Profile.work_contract_reserved`
- `Profile.work_partner_uid`
- `Profile.work_partner_date`

结论：你提到的“打工合约”目前确实还没有真正做完。

### 2.3 打工被牛走的收益结算逻辑没有接到生产流程

`WorkService.resolve_stolen_work()` 已经存在，也有测试覆盖；
但生产代码里 NTR 成功时只是直接清空打工状态，没有调用这条结算逻辑。

这意味着：

- “打工中被牛”的结算分支并没有完整落地
- 目前只是“状态被打断”，没有完整体现“被截胡/被抢收益”的玩法闭环

### 2.4 新手引导 / 周惊喜宝箱只有配置，没有完整玩法闭环

当前只看到配置与数据字段：

- `weekly_surprise_box`
- `newbie_guide`
- `newbie_guide_claimed`

但没有形成清晰的玩家入口、领奖命令或完整展示反馈。

### 2.5 旧占位 service 文件仍未清理

当前仓库里还保留两个未接线的占位 service：

- `app/services/intimacy_service.py`
- `app/services/ntr_service.py`

这两个文件不会影响现有玩家功能，因为实际逻辑都已经进了 `OwnershipService`；
但它们会误导后续维护者，以为系统还有另一套未启用主实现。

### 2.6 NTR 概率流水线仍有预留因子未真正落地

当前 `OwnershipService.try_ntr()` 里已经有完整的概率流水线骨架，但其中仍有明确占位：

- 魅力加成
- 连续被牛惩罚

这意味着当前 NTR 系统虽然可玩，但并不是“策划上所有概率修正项都已完全兑现”的状态。

### 2.7 第二波预留道具/字段仍未形成真实玩法

除打工合约外，当前还存在这类“数据已预留，但玩家端还没有闭环”的内容：

- `insurance_card`
- `work_contract_reserved`
- `work_partner_uid`
- `work_partner_date`
- `intimacy_decay`

这些不是 bug，但确实属于“半接线预留”。

## 3. 已可用但仍值得继续完善的地方

### 3.1 主老婆概念刚补齐，仍需要持续做“无感引导”

现在已经支持：

- `老婆 切换 <编号>`
- `查老婆` / `面板` 用 `👑` 标识主老婆

但玩家理解成本仍可继续降低，比如：

- 首次抽到第二个老婆时主动提示“你现在可以用 `老婆 切换 2` 改主老婆”
- 在 `PK` / `打工` / `互动` 失败提示里更明确地提醒“默认使用主老婆”
- 给 `面板` 增加“当前主老婆名称”专门字段，而不是只靠图标

### 3.2 “老婆是独立单位”这个设计还没完全统一

目前支持按编号指定的玩法：

- `牛老婆`
- `锁定` / `解锁`
- `PK`
- `打工`
- `切换`

但仍默认只作用于主老婆的玩法包括：

- `老婆 摸头`
- `老婆 送礼`
- `老婆 对话`
- `老婆 约会`
- `换老婆`
- `交换老婆`

如果目标是“每个老婆都是独立可单选单位”，那下一阶段应统一这些命令的编号语义。

### 3.3 玩家对规则的理解还可以继续加强

当前帮助已经比之前清楚很多，但还不够像一份“新玩家上手说明”。

后续可补：

- 一个真正的“新手三步走”帮助段落
- “推荐命令顺序”
- “主老婆是什么 / 为什么要切换 / 默认行为是什么”专题说明
- “打工、PK、NTR、亲密度”之间的关系图

### 3.4 仍有一些“代码里有，玩家感知不到”的能力

这类内容会让代码看起来比玩家体验更完整：

- `resolve_stolen_work()` 已有实现和测试，但玩家流程没接通
- `weekly_surprise_box` / `newbie_guide` 只有配置侧与数据侧，没有清晰命令入口
- 复仇相关扩展道具（如 `revenge_token`）已经接入部分逻辑，但整体规则仍不够显性

这类项建议优先决定：

- 要么真正开放给玩家
- 要么先从帮助/文档中隐藏成内部预留

### 3.5 交互默认规则仍有“知道代码的人才懂”的问题

即便现在帮助比之前清楚，仍存在一些默认规则需要玩家自行推断：

- 不写编号时默认作用于主老婆
- 某些命令支持编号，某些命令不支持
- 被牛、打工、PK、主老婆切换之间的状态联动没有集中说明

这类问题不一定要靠新增功能解决，很多可以靠：

- 首次触发时的新手提示
- 更具体的失败文案
- `面板` 中增加“当前默认对象”说明

## 4. 建议优先级

### P0：优先立即做

1. 接通“打工中被牛”的正式收益结算流程
2. 继续强化主老婆概念的玩家引导
3. 清理或明确标注未接线占位 service，避免维护误判

### P1：下一阶段最值回票价

1. 把 `摸头 / 送礼 / 对话 / 约会` 改成支持编号
2. 让“老婆是独立单位”的规则在所有核心玩法里一致
3. 为新玩家补一版更强的帮助与引导文案
4. 决定 `列表 / 查` 这两个分组命令到底要不要正式开放

### P2：补完整个 Phase 4 第二波预留

1. 打工合约
2. 打工搭档
3. 亲密度衰减
4. 周惊喜宝箱
5. 新手引导奖励闭环
6. 保险卡等第二波预留道具是否启用

### P3：最后再做的边缘项

1. `老婆 列表` / `老婆 查` 是否真的还有必要保留独立命令
2. 未使用预留字段、配置和占位文件的彻底收口

## 5. 推荐下一步执行顺序

推荐按下面顺序推进：

1. 修“打工被牛”正式结算链路
2. 统一编号语义到亲密度互动命令
3. 做一版真正的新手帮助 / 新手引导
4. 决定哪些“预留但未开放”的能力要正式上线，哪些先收口隐藏
5. 再决定是否正式开启“打工合约”这一批第二波玩法

## 6. 如果要“全做完”，建议按这份范围理解

如果你的意思是把 `NEXT_GOALS` 写成“所有还能继续做的点都列出来”，当前建议按下面三类理解：

### 6.1 真正没做完的

- `老婆 列表`
- `老婆 查`
- 打工合约 / 打工搭档 / 亲密度衰减
- 周惊喜宝箱 / 新手引导奖励闭环
- 打工中被牛的正式收益结算链路
- NTR 概率流水线中仍是占位的修正因子

### 6.2 已能玩，但规则还没完全统一的

- 编号化互动命令
- 主老婆概念玩家引导
- 默认行为说明
- 独立单位化的一致性

### 6.3 技术债 / 维护债

- 未接线 service 清理
- 预留字段与配置的去留决策
- 帮助/README/规划文档长期一致性维护

## 7. 一句话判断

如果只问一句：

> 这个插件现在能玩吗？

答案是：**能玩，而且主流程已经齐了。**

> 那还有没有明显没做完的地方？

答案也是：**有，尤其是“打工第二波”和“老婆独立单位化的一致性”这两块。**
